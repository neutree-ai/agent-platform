前面 6 章讲的都是单个 Agent 的事——怎么配、怎么调、怎么触发、怎么协作。当这些手艺活做完之后，新的问题会冒出来：

- 你调好的 Prompt，怎么让其他人也用上？
- 你团队里的 5 个 Agent，怎么保持配置一致？
- 一个 Agent 出问题了，怎么把这次会话发给同事看？
- 100 个 Workspace 堆在列表里，怎么找到你要的那个？

这一章讲 Neutree Agent Platform（NAP）里所有跟"复用、共享、组织"有关的能力。

## 资源库三件套：Prompts、Skills、模板

**资源库** app（`⌘K` → **资源库**）里有三类东西，每一类都解决一种复用问题。

### Prompts

把一段反复使用的系统 Prompt 存到资源库里。其他 Workspace 可以**引用**这份 Prompt，引用关系建立后：

- 资源库里 Prompt 一更新，所有引用方**自动同步**
- 想偏离基准的 Workspace 可以"覆盖"，覆盖后就不再跟随更新（也可以随时恢复跟随）

什么时候把一个 Prompt 抽到资源库：

- **多个 Workspace 用同一套行为** ——典型场景是团队规范化某类 Agent
- **同一个 Prompt 在迭代** ——你不希望每次改完都手动同步到 5 个地方
- **想做版本管理** ——资源库 Prompt 支持多版本，方便回溯和对比

如果只有一个 Workspace 在用，没必要抽出来。等真的有第二个使用者再说。

### Skills

Skill 也存在资源库里，机制类似。[定义 Agent 行为](/zh-cn/guides/3-agent-behavior/#skills可复用的特长)里讲过怎么在 Workspace 里**启用**一个 skill。这里讲怎么**创建**一个：

在 **资源库** app 里切到 **Skills**，新建一个。可以选两种方式：

- **上传压缩包** ——把 `SKILL.md` 加上若干工具脚本打成 zip 上传
- **从 Git 仓库导入** ——指定一个仓库地址和路径，平台会拉下来。仓库更新后可以重新同步

Skill 的内容结构是约定的：

```
my-skill/
├── SKILL.md       # 描述这个 skill 是干什么的、提供哪些命令
└── scripts/       # 工具脚本
    └── ...
```

`SKILL.md` 是 Agent 真正读到的内容——它告诉 Agent 这个 skill 提供什么能力、怎么用。写好 `SKILL.md` 是写好 skill 的核心。

### 模板

模板是**完整的 Workspace 配置快照**——模型、Prompt、Skills、MCP、设置、资源全部打包。从模板创建的 Workspace 直接拥有这一整套配置。

什么时候把一个 Workspace 抽成模板：

- **要批量创建同类 Agent** ——比如让团队每个人都有一个"翻译 Agent"
- **想给新成员一份开箱即用的起点** ——他们只需要从模板创建，不需要从零配
- **配置在迭代** ——模板更新后，绑定它的 Workspace 可以一键升级

在 Workspace 的 **设置** app 的 **常规** 区域点 **另存为模板**。可以选择是否把当前 Workspace **绑定**到这个新模板（绑定后能跟随更新）。

### 模板和资源库 Prompt 的关系

两者不冲突，配合起来才完整：

- **模板** ——管整个 Agent 的"默认人设"
- **资源库 Prompt** ——管 Prompt 这一项的细粒度迭代

很多团队的实践是：模板里的 Prompt 字段引用一份资源库 Prompt。这样模板提供整体配置基线，Prompt 可以独立迭代，更新后所有从模板创建的 Workspace 都跟着收到新 Prompt。

## 标签：组织你的 Workspace 列表

Workspace 攒到几十个之后，找一个就开始难。**标签** 是 NAP 提供的轻量分组工具。

在首页的 **标签** app（`⌘K` → **标签**）里创建和管理标签。给 Workspace 打标签在它的 **设置** → **常规** 里。顶部菜单栏的 workspace 切换器支持按标签过滤列表。

### 标签设计建议

- 按**用途** ——`production`、`staging`、`experiment`
- 按**团队** ——`frontend`、`backend`、`data`
- 按**状态** ——`active`、`archived`、`review`

颜色用来直观区分。标签筛选是 OR 逻辑——选中多个时匹配任一即显示。

## 共享会话

调试 Agent 时经常需要把一次会话发给同事看：你看 Agent 在某一步走错了，想让同事帮你诊断。

用会话上的分享按钮（**对话** 和 **会话历史** 里都有）。生成一个公开链接，任何人打开都能看到这个 Session 的完整对话——消息、工具调用、文件操作全部可见。

适合的场景：

- **求助** ——把出问题的会话发给团队里更熟的人
- **演示** ——给业务方看一次完整的端到端流程
- **复盘** ——某次 Agent 表现特别好或特别糟，存档备查

注意分享出去的内容是**公开**的——不要分享包含敏感信息的会话。

## Workspace 可见性与团队协作

[多 Agent 协作](/zh-cn/guides/6-compose-agents/#visibility)讲过 Workspace 的 Visibility 影响**谁能调用它**。同样的字段也影响**谁能在自己的列表里看到它**：

- **Private** ——只有你自己能看
- **User** ——你自己看得到（不出现在他人列表）
- **Public** —— 实例内所有用户可见

资源库里的 Prompts、Skills、模板遵循平台统一的三层 scope —— **Private** / **Team** / **Public**。Public 适合全实例共享的能力，Team 限定给选定团队，Private 适合个人使用或试验阶段。

### 一个常见的团队模式

1. **个人空间试错** ——在 Private 的 Workspace 里调 Prompt、试 skill，自由实验
2. **稳定后抽到资源库** ——把 Prompt 放到 Public Prompts，把 skill 放到 Public Skills
3. **沉淀成模板** ——把成熟的 Workspace 配置存成 Public 模板，让团队每个人都能一键创建
4. **持续迭代** ——通过资源库的"更新自动同步"机制，配置改进自动下发

这个流程把"个人探索"和"团队收益"连起来——一个人花时间调好的东西，全团队都能用上。
