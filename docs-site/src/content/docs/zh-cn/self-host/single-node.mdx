在单节点 k3s 集群上安装 Neutree Agent Platform —— 让完整平台跑起来的最快方式。单节点 profile 与完整 profile 一致，只是 PostgreSQL 为单实例、共享存储由集群内 NFS server 提供。所有镜像都从公共 registry（`ghcr.io` / `docker.io` / `registry.k8s.io`）拉取，节点需要能访问互联网。

推荐配置：**8 vCPU / 32 GB / 200 GB 磁盘**，约可支撑 **10 个并发 workspace**。需要 HA 或横向扩展时，走[生产部署](/zh-cn/self-host/)。

> 完全隔离（air-gapped）的离线安装是可行的，但尚未打包进开源发行版 —— 有需要请提 [GitHub issue](https://github.com/neutree-ai/agent-platform/issues)。

## 一行命令安装

### 裸 Linux 主机（无 Kubernetes）

**不需要**预装 k3s，脚本会自动安装。以 root 执行：

```bash
curl -sfL https://nap.docs.neutree.ai/get.sh | sudo sh -
```

脚本会在缺失时安装 k3s，生成全部 secret，自动探测节点 IP，随机生成 admin 密码，完成后打印登录 URL 和凭据。配置落在 `/opt/nap/values.env`；重复执行同一行命令即为原地升级。

常用覆盖方式：

```bash
# 自行指定节点 IP / admin 密码
curl -sfL https://nap.docs.neutree.ai/get.sh \
  | sudo NAP_HOST=192.168.1.10 NAP_ADMIN_PASSWORD=your-password sh -

# 只生成 values.env —— 人工确认或修改后，去掉该 flag 重新执行即开始安装
curl -sfL https://nap.docs.neutree.ai/get.sh \
  | sudo sh -s -- --prepare-only
```

`get.sh --help` 列出全部 flag（`--version=` 固定发布版本、`--dir=` 更换安装目录）。

### 已有 Kubernetes 集群

使用当前 kubeconfig（`$KUBECONFIG`，默认 `~/.kube/config`）。有两样东西无法自动探测、必须显式给出：用户访问平台的 host，以及 RWX 存储 —— 外部 NFS export 或集群里已有的 RWX StorageClass，二选一：

```bash
# 使用外部 NFS server
curl -sfL https://nap.docs.neutree.ai/get.sh \
  | sh -s -- --k8s --host=<ip-or-hostname> --nfs-server=<ip> --nfs-path=</export/path>

# 使用已有的 RWX StorageClass
curl -sfL https://nap.docs.neutree.ai/get.sh \
  | sh -s -- --k8s --host=<ip-or-hostname> --storage-class=<rwx-storageclass>
```

这会以默认配置安装完整 profile（3 副本 PostgreSQL）；加 `--prepare-only` 可先审阅 `/opt/nap/values.env` 再安装。生产级的规划 —— 架构、Ingress、容量 —— 请走[生产部署指南](/zh-cn/self-host/)。

## 手动部署

与一行命令等价的分步流程，便于逐项掌控。在一台已就绪的 k3s 主机上执行，其 kubeconfig 位于 `/etc/rancher/k3s/k3s.yaml`（单节点示例中的默认路径）。

### 1. 准备 values.env

```bash
cd self-host
cp values.env.single-node.example values.env
./gen-secrets.sh        # fills random JWT / PG / TURN / SANDBOX secrets
vi values.env           # set TOS_HOST + ADMIN_PASSWORD
```

`values.env.single-node.example` 已经内置了单节点默认值，所以你只需要改两处：

- `TOS_HOST` —— 节点对外可达的 IP；pod 和登录入口都使用它
- `ADMIN_PASSWORD` —— 初始 admin 密码

其余所有项（PG 副本数、storage class、NFS 后端）都已按单节点形态预置。

### 2. 安装

```bash
./install.sh --profile=single-node
```

`install.sh` 会先拉起集群内 NFS server（`nap-nfs-server` pod，以 `local-path` 为后端），从公共源安装 CloudNativePG operator 和 NFS subdir provisioner，渲染并应用 manifest，然后初始化（seed）admin 用户、OAuth client 和 MCP 目录。所有镜像都从公共 registry 拉取。

### 3. 登录

```
http://<TOS_HOST>:30080
admin / <ADMIN_PASSWORD>
```

## 与标准部署的关键差异

下表仅列出与[部署指南](/zh-cn/self-host/)默认值不同的配置；其余沿用标准默认：

| 字段 | 标准默认 | 单节点 |
| --- | --- | --- |
| `DEPLOY_PROFILE` | `multi-node`（隐式） | `single-node` |
| `PG_INSTANCES` | 3 | 1 |
| `PG_STORAGE_CLASS` | 任意 RWO CSI | `local-path` |
| `AGENT_STORAGE_CLASS` | NFS / RWX CSI | `local-path` |
| `NFS_SERVER` / `NFS_PATH` | 外部 NFS | 集群内 NFS pod（`nap-nfs-server`），由 `install.sh` 计算得出 |

这些都已写入 `values.env.single-node.example`，无需手动填写。

:::caution[主机必须启用 NFS 内核模块]
集群内 NFS 使用主机内核：`nap-nfs-server` pod 启动时挂载 `nfsd`，nfs-subdir provisioner 挂载时需要 `nfs` 客户端模块。若模块缺失，`nap-nfs-server` pod 起不来，AFS PVC 会一直 `Pending`。

检查是否已加载：

```bash
lsmod | grep -E 'nfs|nfsd'
```

如果没有任何输出，手动加载两者：

```bash
sudo modprobe nfs
sudo modprobe nfsd
lsmod | grep -E 'nfs|nfsd'   # confirm loaded
```

如果 `modprobe` 报 `module not found`，说明内核缺少 NFS 模块包 —— 先安装它（例如 `apt-get install linux-modules-extra-$(uname -r)`）或换成带 NFS 支持的内核镜像，再重新执行安装。
:::

## 注意事项

- **数据在节点上** —— PVC（含 AFS 共享文件）使用 `local-path`，绑定节点磁盘；重要数据请做外部备份
- **安装器在节点本机运行** —— 单节点 profile 把镜像拉到本地集群
- **升级会重启 pod** —— 镜像滚动期间有短暂窗口；流程与[标准升级](/zh-cn/self-host/#upgrade)一致：重跑一行命令或 `./install.sh --profile=single-node`
